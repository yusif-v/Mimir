# Mimir v0.2 — Finish the Core Loop

**Date:** 2026-06-18
**Status:** Approved design, ready for implementation plan

## 1. Goal & Scope

Make `run → capture → case → timeline` work end to end. After this release, an
analyst can: open a case, drop evidence in, `run hash`/`file`/`strings` (or a
Docker tool) against it, have every run's output saved to the case and recorded
as a timestamped timeline event, and view the whole investigation
chronologically with `timeline`.

**In scope:**
- Timeline data model (append-only JSONL + in-memory cache)
- Run-recording wiring (output capture + timeline event on every run)
- Three native built-in tools: `hash`, `file`, `strings`
- `timeline` command

**Out of scope (tracked separately / deferred):**
- Event-bus handler unsubscribe (`events.Bus.Off()`) — known gap, optional standalone fix
- Evidence management commands (add/tag/hash workflow) — v0.3
- Case export to JSON archive — v0.3
- Timeline filtering by type — v0.3

## 2. Architecture Overview

```
cmdRun(tool, args)
  │
  ├─ Runner.Run(...)            → Result{stdout, stderr, code, started, finished}
  │     (Docker path | local path | NEW: builtin path)
  │
  └─ on the open case:
        ├─ OutputCapture.Record(...)   → writes output/<tool>_<ts>.txt (returns path)
        ├─ Case.AppendEvent(TimelineEvent{...})  → appends to timeline.jsonl + in-mem cache
        └─ Case.Save()                  → case.json metadata
```

Recording is synchronous and lives inside `cmdRun`. The event bus keeps emitting
`ToolFinished` / `OutputCaptured` for observers (plugins, future Heimdrall/SIEM
hooks), but the core loop no longer depends on the bus.

**Rationale for direct-call over event-driven recording:**
1. Recording *is* the loop's deliverable, not a side effect.
2. Forensic correctness needs synchronous error handling — a failed save must
   surface to the analyst, not be swallowed by a fire-and-forget handler.
3. `casePath` is already in hand in `cmdRun`; event-driven would force it into
   every event payload.
4. Straight-line testability (run → assert file + timeline entry).
5. Avoids coupling the feature to the unfixed unsubscribe gap.

## 3. Timeline Data Model

New type in `internal/cases`:

```go
type TimelineEvent struct {
    Type      string         `json:"type"`       // "tool_run" | "note" | "case_opened" | "case_closed"
    Timestamp string         `json:"timestamp"`  // RFC3339
    Payload   map[string]any `json:"payload"`
}
```

Payloads:
- `tool_run`: `tool` (string), `args` ([]string), `return_code` (int),
  `duration_ms` (int64), `output_file` (string, relative to case dir),
  `success` (bool). On failure, also `stderr` (string).
- `note`: `author` (string), `content` (string).
- `case_opened` / `case_closed`: empty payload (timestamp carries the meaning).

**Storage — the "both" decision:**
- `timeline.jsonl` in the case dir is the **append-only source of truth**. One
  JSON object per line. Never rewritten — forensic integrity.
- `Case.events []TimelineEvent` is an **in-memory cache**, populated by streaming
  `timeline.jsonl` on `LoadCase` / open. NOT serialized into `case.json`.

New methods on `Case`:
- `AppendEvent(ev TimelineEvent) error` — marshal one line, append to
  `timeline.jsonl` (open with `O_APPEND|O_CREATE|O_WRONLY`), and append to the
  in-memory cache.
- `Timeline() []TimelineEvent` — return the in-memory cache.
- `loadTimeline() error` — stream `timeline.jsonl`, skip+log corrupt lines,
  populate cache. Called from `LoadCase` (missing file is fine → empty timeline).

**`ToolsUsed`:** kept as the deduped "tools touched in this case" summary in
`case.json` (cheap, still used by `status`). Per-run detail now lives in the
timeline.

**`AddNote`:** additionally appends a `note` `TimelineEvent` so notes interleave
chronologically with tool runs.

## 4. Recording Path (Wiring)

`cmdRun` after `Runner.Run`:
1. Print stdout as today.
2. If a case is open:
   a. `OutputCapture.Record(tool, output, casePath)` → returns
      `(outputPath string, err error)` (signature change: previously returned
      only `error`).
   b. Build the `tool_run` `TimelineEvent` (duration from `Result.Duration()`,
      return code, the relative output path, success flag) and
      `Case.AppendEvent(...)`.
   c. `AddToolUsage` + `Save` as today.
3. If `Record` or `AppendEvent` errors, print a warning to the analyst (do NOT
   swallow), but still report the run result.

**Failed runs are recorded too:** a run where `!Result.Success()` is still
written as a `tool_run` event with `success:false` and `stderr` captured — an
analyst's failed attempts are part of the investigation record.

## 5. Built-in Tools (Native Go)

New package `internal/builtins`. Each tool is a function with signature:

```go
func(args []string) (stdout string, err error)
```

Tools:
- **`hash <file>`** — `crypto/md5`, `crypto/sha1`, `crypto/sha256`; prints all
  three digests. Streams the file (`io.Copy` into a `io.MultiWriter` of the
  hashers) so large evidence does not exhaust memory.
- **`strings <file> [-n N]`** — scans bytes for runs of ≥ N printable ASCII
  (default N=4), prints each run on its own line. Read in chunks; handle runs
  spanning chunk boundaries.
- **`file <file>`** — magic-byte sniffing via a small built-in signature table
  (PE/ELF/Mach-O, ZIP/PDF/PNG/JPEG/GZIP, …) with `http.DetectContentType` as a
  fallback. Prints the detected type.

**Dispatch:** `cmdRun` checks a builtins registry *first*; if the name matches a
builtin, the `Runner` executes it via a new `runBuiltin` path that produces a
normal `Result` (StartedAt/FinishedAt set, Stdout/Stderr/Error populated). This
keeps capture + timeline recording identical across Docker / local / builtin
paths.

**Status & listing:** builtins are always `StatusReady` (no Docker, no PATH
dependency). They appear in the `tools` view under a "Built-in" section with a
`builtin` badge, consistent with the existing two-section tools view, without
going through the Docker catalog.

## 6. `timeline` Command

New `timeline` command in `internal/shell`:
- No case open → error `"no case is open"`.
- Renders `Case.Timeline()` chronologically. Per line:
  `HH:MM:SS  ICON  summary`.
  - `tool_run` → `▶ run <tool> <args> → code N (Xms)  [output/<file>]`,
    colored green for success / red for failure.
  - `note` → `✎ note: <content>`.
  - `case_opened` / `case_closed` → markers.
- Flag: `timeline -n <N>` tails the last N events. (Filtering by type → v0.3.)

## 7. Error Handling

- `AppendEvent` write failure → returned to caller; `cmdRun` prints a warning but
  the run result still shows. Timeline integrity issues are loud, never silent.
- `loadTimeline` with a corrupt `timeline.jsonl` line → skip the bad line, log a
  warning, keep the rest. One bad append must not brick case loading.
- Built-in tools validate that the target path exists and is a regular file
  before processing; clear errors otherwise.
- `Record` returning an error must not lose the analyst's view of the run output
  (already printed to stdout in step 1).

## 8. Testing

- **cases:** append N events → reload from disk → cache matches; corrupt-line
  tolerance (one bad line skipped, rest intact); `AddNote` appends to both
  `Notes` and the timeline; missing `timeline.jsonl` → empty timeline, no error.
- **builtins:** known fixtures → known md5/sha1/sha256; `strings` extracts
  expected runs honoring `-n`; `file` identifies PE/ELF/PNG/ZIP fixtures;
  non-existent / directory targets error clearly.
- **shell:** `run <builtin>` against a temp evidence file in a temp case →
  asserts output file written under `output/` AND a `tool_run` timeline event is
  present with correct payload; a failed run is still recorded with
  `success:false`.
- Existing suite stays green.
